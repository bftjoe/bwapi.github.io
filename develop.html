Coding Standards

Spacing

Spacing is important. Don't be afraid of spacing.

  1. Use double-spaces instead of tabs. You should be able to convert tabs to spaces automatically in your editor's settings. Spaces will ensure that code will always be aligned correctly regardless of tab settings.
  2. Always indent for each scope level.
  3. Always put a space between all arithmetic, binary, and conditional operators.
  4. Don't put a space between function names and the open parenthesis.
  5. Put a space after a comma.

Examples of spacing

// bad, can't tell if assignment is a typo, and
// there is a space between the member function and its parameters
if(Broodwar->canMake (UnitTypes::Terran_Marine, builder))
reservedMinerals=-UnitTypes::Terran_Marine.mineralPrice();
// good, the arithmetic is clear
if(Broodwar->canMake(UnitTypes::Terran_Marine, builder))
reservedMinerals = -UnitTypes::Terran_Marine.mineralPrice();
// bad, the scope is not indented, no space following the comma
if ( Broodwar->canMake(UnitTypes::Terran_Marine,builder) )
reservedMinerals = -UnitTypes::Terran_Marine.mineralPrice();
// good, alternative spacing style for if statements
if ( Broodwar->canMake(UnitTypes::Terran_Marine, builder) )
reservedMinerals = -UnitTypes::Terran_Marine.mineralPrice();

Formatting

  1. Use ANSI style braces.
  2. Break up large single-line if statements to be multi-line if statements.
  3. Format constructor initializer lists as follows.
Player::Player()
:unitCount(0)
,pFirstUnit(NULL)
,name("")
,isWinner(false)
{
// additional constructor code
}

Naming

  1. Try to use meaningful names. If a variable name cannot be resolved by another project member, then the variable needs to be renamed.
  2. In local looping scopes, single-letter variable names are generally used as follows:
    • b for Bullet
    • f for Force
    • p for Player
    • r for Region
    • u for Unit
    • i for iterator/index
  3. Member variables and member function names should be in lower camel case. Examples: getUnitsInRectangle, wasSeenByBWAPIPlayer.
  4. Constants and macros should (in most cases) be in ALL CAPS. Example: PLAYER_COUNT
  5. Use upper Camel Case for class names and namespaces. Do not use underscores in class names. Instead put it inside of a namespace.
  6. Enumerations should be in upper Camel Case.

Hacking

  1. Offsets should be in hexadecimal, and all offsets are to be separate from the coding itself. Example: 0x00408CF0
  2. Avoid using inline assembly unless it is impossible to do so.
  3. Perform version checking before making code patches, to ensure that BWAPI is compatible with all versions of Starcraft: Broodwar.

Documentation

  1. Documentation should follow Doxygen style formatting.
  2. Use the triple slash (///) format to ensure that the documentation displays correctly in Microsoft Visual Studio's intellisense tooltips.
  3. Use @ for doxygen commands.
  4. Use @see instead of @sa to refer to other functions or classes.
  5. When introducing a new function, include a @since tag with the revision number.
  6. Include any @param, @returns, and @retval where necessary.
  7. Try to include as much details on the function as possible. Some redundancy in documentation can express clarity to users.
  8. Include example code if possible.
  9. Wrap lines before the 100th column.
  10. When specifying @param, also specify if the argument is optional. Present an indented description on the next line, and also specify its default value if applicable.
  11. If a piece of code is incomplete or temporary, then label it as such.

Examples of documentation

virtual bool isVisible(Player* player = nullptr) const = 0;

Language Features

  1. Use the new nullptr keyword instead of NULL. This is to conform with the new standard.
  2. Create move constructors if appropriate (especially when memory allocation is involved). This is to improve performance.
  3. Use the const keyword where appropriate.

Creating a Release

Building Binaries (Windows)

Requirements

  • Microsoft Visual Studio Express 2013 for Windows Desktop *(Note: Must say DESKTOP)*

Instructions

  1. Open the project solution in Visual Studio.
  2. Ensure that you have permission to write to the Starcraft directory. This can be done by modifying the permissions of the directory or installing the program in the user space.
  3. Build for both DEBUG and RELEASE.

Generating Documentation (Any OS)

Requirements

  • Doxygen (1.8.6 or newer)
  • Graphviz (For prettier diagrams)
  • Python optional (For one-command generation)

Instructions

With Python

  1. Navigate to bwapi/Documentation.
  2. Change the version information in Doxyfile.
  3. Run python generate.py

Without Python

  1. Navigate to bwapi/Documentation.
  2. Run git clone git@github.com:bwapi/bwapi.wiki.git (or pull updates into the repository)
  3. Change the version information in Doxyfile.
  4. Run doxygen.

Documentation should be created in bwapi/Release_Binary/documentation and ready for packaging. The generated log messages should contain no errors.

Building Installer (Windows)

In order to build the installer, you must first complete the Building Binaries and Generating Documentation steps.

Requirements

  • Inno Setup (Get the QuickStart package with Inno Script Studio)

Instructions

  1. Navigate to bwapi/Installer.
  2. Open installer.iss.
  3. Change the version information.
  4. Click on Project and go to Compile.
